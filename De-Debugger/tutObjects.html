Why do we need Object Orientatted Programming (OOP)?

OOP helps to keep the C++ code DRY "Don't Repeat Yourself", and makes the code easier to maintain, modify and debug. OOP makes it possible to create full reusable applications with less code and shorter development time.

Classes

In C++, a class is a user-defined data type that encapsulates information and behavior about an object. It serves as a blueprint for future inherited classes.

A class consists of data members (attributes) and member functions (methods). The data members represent the state of an object, while the member functions represent its behavior.

In the above example, we defined a class called MyClass with two data members (myInt and myString) and one member function (printMessage()). We can create objects of this class and use its member functions to perform actions and access its data members to retrieve its state.


Class Members

A class is comprised of class members:

  • Attributes, also known as member data, consist of information about an instance of the class.
  • Methods, also known as member functions, are functions that can be used with an instance of the class.

Access Control Operators

C++ classes have access control operators that designate the scope of class members:

  • `public` members are accessible everywhere.
  • `private` members can only be accessed from within the same instance of the class or from friends classes.

Polymorphism

Polymorphism is an important concept in object-oriented programming. It means "more than one form" — the same entity (function or operator) can operate differently under different situations.

There are two types of polymorphism in C++:

  • Compile-time polymorphism: This is also called static (or early) binding.
  • Runtime polymorphism: This is also called dynamic (or late) binding.

    • Compile-time Polymorphism

    Compile-time polymorphism is perfect for function overloading as well as operator overloading.

Since the call is determined during compile time, it is called compile-time polymorphism.


    Runtime Polymorphism

    Function overriding is an example of runtime polymorphism.

    When a child class declares a method that is already present in the parent class, it is called function overriding because the child class overrides the parent class.

    In the case of function overriding, there are two definitions of the same function, one in the parent class and one in the child class. Since the call to the function is determined at runtime to decide which definition of the function is to be called, it is called runtime polymorphism.


    Why do we need Polymorphism?

    Polymorphism in C++ allows us to reuse code by creating one function that's usable for multiple uses. We can also make operators polymorphic and use them to add not only numbers but also combine strings. This saves time and allows for a more streamlined program.

    Passing objects by value in C++

    Passing objects by value in C++ can be problematic for several reasons:

    • Performance: Passing objects by value involves making a copy of the object, which can be time-consuming and memory-intensive, especially for large objects. This can result in slower code and higher memory usage.
    • Object slicing: When passing an object of a derived class by value as a parameter to a function that expects an object of its base class, only the base-class portion of the object is passed. This is called object slicing, and it can lead to unexpected behavior and errors.
    • Object ownership: If the object being passed is owned by the caller of the function, passing it by value may result in a copy being made and unnecessary memory usage. If the object is owned by the called function, passing it by value may result in a copy being made and the original object being left unchanged.
    • Object state: If the object being passed has mutable state, passing it by value may result in unintended changes to the original object, as the copy made by the function may be modified independently.

    To avoid these problems, it is often better to pass objects by reference or pointer instead of by value, depending on the situation. This avoids unnecessary copies and allows the function to operate on the original object without modifying it.


    Memory leaks

    Memory leaks occur when dynamically allocated memory is not properly released when it is no longer needed. This can happen with objects if the programmer forgets to call the destructor or delete the object when it is no longer needed.


    Null pointer errors

    Null pointer errors occur when a program attempts to access a pointer that is null or uninitialized. This can happen with objects if the programmer forgets to initialize a pointer variable or does not check for null pointers before accessing them.


    Access violations

    Access violations occur when a program attempts to access memory that it does not have permission to access. This can happen with objects if the programmer accesses private or protected members of a class from outside the class, or if the program uses an object after it has been destroyed.


    Object slicing

    Object slicing occurs when a derived class object is assigned to a base class object, resulting in the loss of the derived class's additional data members and member functions. This can happen when objects are passed by value or assigned to variables of a different type.
